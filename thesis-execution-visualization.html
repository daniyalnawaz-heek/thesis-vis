SKOS VOCABULARY INTEGRATION - Phase 2.3

SKOS Concepts vs OWL Equivalence:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Why SKOS Instead of owl:sameAs?
────────────────────────────────────
owl:sameAs Problems:
  ✗ Causes transitive inferences (if A=B and B=C, then A=C automatically)
  ✗ Merges conflicting definitions
  ✗ Loses framework-specific context
  ✗ Example: MITRE severity≠OWASP severity, but owl:sameAs merges them

SKOS Solution:
  ✓ Preserves framework-specific context
  ✓ Allows nuanced mapping (exact vs close match)
  ✓ Maintains provenance information
  ✓ No unwanted transitive inferences


SKOS MAPPING VOCABULARY:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. skos:exactMatch
   Meaning: Same phenomenon, equivalent concept

   Example:
   mitre:T1921 skos:exactMatch owasp_ml:ML04
   (Both describe Data Poisoning - same thing)

   Framework-specific details preserved:
   mitre:T1921 sec:severity 0.8
   owasp_ml:ML04 sec:severity 0.9
   (Different severity assessments maintained)


2. skos:closeMatch
   Meaning: Related but not identical concepts

   Example:
   mitre:T1594 skos:closeMatch owasp_ml:ML05
   (Model Extraction vs Model Stealing - similar but different terms)

   Rationale: Both involve unauthorized model access,
   but different threat vectors


3. skos:relatedMatch
   Meaning: Loosely related concepts

   Example:
   mitre:SupplyChainCompromise skos:relatedMatch owasp_llm:LLM06
   (Supply chain threats related to LLM vulnerabilities)


4. skos:narrowMatch / skos:broadMatch
   Meaning: Hierarchical relationships

   Example:
   general:DataManipulation skos:broadMatch specific:DataPoisoning
   (Data Poisoning is a type of Data Manipulation)

The paper compares:
1. OWL (Web Ontology Language) - for formal reasoning and class hierarchies
2. SKOS (Simple Knowledge Organization System) - for vocabulary mapping and thesaurus

Key Findings:
- OWL's owl:sameAs causes problems in large knowledge bases (DBpedia, Wikidata)
  Problem 1: Transitive inferences (A=B, B=C → A=C automatically, even if wrong)
  Problem 2: Ontological confusion (conflicting definitions get merged)

- SKOS mapping relationships are safer for multi-source integration:
  skos:exactMatch = strong equivalence without automatic inference
  skos:closeMatch = semantic relationship without claiming identity
  skos:broadMatch, skos:narrowMatch = hierarchical relationships

THE PROBLEM WITH owl:sameAs:

Transitive Inference Problem:
  If: A owl:sameAs B  AND  B owl:sameAs C
  Then: A owl:sameAs C  (automatic inference)

Real World Impact - Your Thesis Example:
  MITRE "Data Poisoning" owl:sameAs OWASP "Data Poisoning"
  OWASP "Data Poisoning" owl:sameAs "Untrusted Data"

  Unwanted Result: MITRE "Data Poisoning" owl:sameAs "Untrusted Data"
  Problem: These are NOT the same! Now MITRE's threat is merged with OWASP's

More Serious Problem - Numerical Conflicts:
  MITRE: mitre:T1921 severity 0.8
  OWASP: owasp:ML04 severity 0.9

  With owl:sameAs:
    OWL reasoner sees them as IDENTICAL entities
    Which severity is TRUE? 0.8 or 0.9?
    CONFLICT! Semantic inconsistency

  With SKOS:
    Reasoner understands they refer to SAME CONCEPT
    But properties remain SEPARATE and VALID
    No conflict! Both framework perspectives preserved

Reason from Halpin:
  - owl:sameAs = logical equivalence (too strong for multi-framework)
  - SKOS vocabulary = conceptual equivalence (perfect for your use case)

Your Vocabulary Choices:

1. skos:exactMatch
   Meaning: "Refer to identical threat phenomenon across frameworks"
   Use When: Both frameworks describe same attack/vulnerability

   Meaning: "Related concepts but different execution/context"
   Use When: Frameworks address same threat area differently
   Example: "Model Extraction" closeMatch "Model Stealing"
   (Both steal models but via different methods)

3. skos:broadMatch / skos:narrowMatch
   Meaning: "Hierarchical relationships between concepts"
   Example: "API Misuse" broadMatch "Security Vulnerability"
